For everyone · no coding required

Build WordPress pages by chatting with Claude.

You don't write code. You give Claude a Figma design and a few notes — Claude builds the page section by section, you preview each one, approve it, and move to the next. When you're done, real content editors can edit everything in WordPress like normal.

👋

Welcome — here's what this is

You have: a Figma design from a designer + a list of pages to build.
You want: a real WordPress site that matches the design, where content editors can later change text/images without breaking anything.
The problem: normally you'd hand the Figma to a developer and wait days. Then changes take more days.
What we do instead: you chat with Claude (in this Cowork app), paste a Figma link, and Claude builds the section directly into WordPress. You watch it happen. If something looks off, you tell Claude in plain English, Claude fixes it.

You don't need to write code. You DO need to be able to look at the live preview and tell Claude when something matches the design and when it doesn't. That's the only skill required.

What Cowork is

The desktop chat app where you're reading this. It runs Claude (the AI). You chat with it about your project, it builds files on your computer.

What Claude is

An AI that can read Figma, write WordPress code, and follow the rules that live in this template (the .claude/ folder). You don't talk to it like Google search — talk to it like a junior developer.

What WordPress is

The website platform underneath. Once Claude builds a section, the editor (you or the client) can change text/images in WordPress like editing a Word document.

📖

Quick glossary (no jargon survives past here)

Five words to know. Reread this anytime someone says one of these and you go "huh?"

Section

One horizontal band on a page. A "hero section" is the big banner at the top. A "testimonials section" is the row of quotes. We build one section at a time, never the whole page in one go.

Figma

The design tool the designer used. You'll paste links to specific frames inside Figma (one for desktop, one for mobile). The link looks like figma.com/design/abc?node-id=1-2.

Brief

A short text file in briefs/ where you write a few notes about the section: name, special behaviors, anything weird. Claude reads it before building. Most sections need only 5-15 lines.

Static / Dynamic

Two phases. Static = Claude hard-codes the design exactly from Figma so you can see it match perfectly. Dynamic = Claude swaps the text/images for editable fields the client can change later. Always static first, dynamic second.

ACF (Sync changes)

ACF = the plugin in WordPress that creates editable fields. After Claude makes a section dynamic, you go to WP Admin → Custom Fields → Field Groups, you'll see a banner that says "Sync changes" — click it. That's how the new editable fields turn on.

MCP

"Model Context Protocol" — fancy name for the plugins that let Claude talk to other apps like Figma. You install one for Figma so Claude can read your designs. Done once per computer.

🚀

First-time setup on a new project

~10 minutes total. Do these once per new project, never again.

1

Make sure your developer has set up the project

Your dev should have installed WordPress locally (probably with Laragon or Local), generated the theme, and dropped the .claude/ folder inside the theme. If you don't know what that means, ask your dev "is the .claude folder ready?". If yes, continue.

2

Open the project folder in Cowork

In Cowork, look for "Select folder" or the folder-picker icon. Navigate to your project's theme folder (your dev knows the path — something like D:\laragon\www\client-name\wp-content\themes\theme-slug\). Click "Select".

Why: this is how Cowork knows which folder Claude can read and write.

3

Connect Figma to Cowork

Open Cowork's Settings → look for "Connectors" (or "Integrations" or "MCPs"). Find Figma in the list → click "Connect" → log in with the same Figma account that has access to your project designs.

Why: Claude needs Figma access so it can see the designs. You only do this once per computer; it remembers.

More detail if needed: INSTALL-MCPS.md

4

Run /setup-claude

In the Cowork chat box, type:

/setup-claude

Claude will detect things automatically and ask you 2-3 small questions: project name, your Figma file URL (optional but recommended), anything special about the project. Just answer in plain English. ~2 minutes.

5

Finish in WordPress admin (your dev or you)

Open your local WordPress admin (usually http://yourproject.test/wp-admin/) and:

  1. Settings → Permalinks → choose "Post name" → Save
  2. Pages → Add New → title "Home" → on the right, Template = Homepage → Publish
  3. Settings → Reading → Homepage displays = "A static page" → Homepage = Home → Save
  4. Custom Fields → Field Groups → click any blue "Sync changes" notice you see

Why: these are WordPress settings Claude can't change for you (they're outside the theme folder). After this, you're ready to build.

🎉 Setup is done

You now have a personalized README.md file in your project. Open it anytime as a daily reference. Now jump to "A full walkthrough" to build your first section.

💡

The big idea — one section at a time

A web page is just a stack of sections — hero at the top, then "about us", then "services", then "testimonials", then footer. We build them one at a time, never all at once. For each section:

Step 1: STATIC

Claude writes the section with the actual text + images from Figma, hard-coded. You preview it, check it matches Figma pixel-by-pixel. If it doesn't, tell Claude what's off — Claude fixes. Repeat until it looks perfect.

Step 2: DYNAMIC

Once the static looks great, you say "make it dynamic". Claude swaps the hard-coded text + images for editable fields in WordPress. The client (or you) can later change them in WP Admin.

Step 3: SYNC + FILL

You go to WP Admin → Custom Fields → click "Sync changes". The editable fields appear. You fill them in once with the real content. Done — section is live.

⚠️ Why static first, dynamic second

If you skip the static phase and go straight to dynamic, every tiny visual problem becomes 3x harder to fix because you can't tell anymore whether the bug is in the design code or in the editable-field code. Static-first is non-negotiable. Trust the process.

🚶

A full walkthrough — building one section

Pretend you're building the home page hero (top banner). Here's every step — no coding, no file creation.

1

Get the Figma link for the hero frame

In Figma, click the hero frame. Right-click → "Copy link". You'll get something like https://figma.com/design/abc/Project?node-id=1-2. If the designer made a separate mobile frame too, grab that link as well.

One link or two? One link = Claude builds desktop AND auto-makes it responsive (it'll tell you what assumptions it made for mobile so you can confirm). Two links = Claude pixel-matches both desktop and mobile exactly.

2

Paste them in chat with the section name

In Cowork chat, type the section name with @ in front, then paste the Figma link(s):

@home-hero https://figma.com/design/abc?node-id=1-2

Or with both desktop and mobile:

@home-hero https://figma.com/design/abc?node-id=1-2 https://figma.com/design/abc?node-id=1-3

That's it. No files to create. Wait ~30-60 seconds. Claude reads Figma, writes the WordPress code, AND auto-creates a brief file at briefs/home-hero.md for the paper trail (you don't have to do anything with it). Claude replies with what it did + any judgment calls.

Want to add a one-off tweak? Just paste extra text after the URLs:

@home-hero https://figma.com/.../1-2 use one big bg image not per-card, h3 not h2 for card titles
3

Preview in your browser

Open http://yourproject.test/ (your local WP site). Hard-refresh (Ctrl+Shift+R on Windows, ⌘+Shift+R on Mac). You should see the hero section. Compare side-by-side with Figma.

Tip: resize your browser window to phone-width (or use Chrome DevTools' device toolbar) to check the mobile version. If Claude listed "mobile guesses" in the reply, this is when you confirm them.

4

If something's off, tell Claude

In plain English. Examples that work great:

The heading is too small on mobile — Figma shows 32px, the live page looks like 24px.
The CTA button color doesn't match — Figma is more teal, I'm seeing dark blue.
The mobile guesses look wrong — image should be ABOVE the text on mobile, not below.

Claude will fix and reply. Refresh, check again. Repeat until it's right.

5

When the design matches: make it dynamic

Just type:

Make home-hero dynamic

Claude swaps the hard-coded text/images for ACF fields and tells you to sync.

6

Sync ACF in WP Admin

Open http://yourproject.test/wp-admin/ → Custom Fields → Field Groups. You'll see a notice "1 field group has changes available". Click the "Sync changes" link.

Now go to Pages → Home → edit. You'll see new editable fields (heading, subheading, image, CTA). Fill them in, save. The hero now displays your content.

7

🎉 Done — onto the next section

Repeat steps 1-6 for the next section (services, about, etc.). The template understands the workflow, so steps get faster as you go.

✨ Shortcut: /build

Once you trust the workflow, you can chain steps 2 + 4 + 5 in one go:

/build home-hero https://figma.com/.../1-2 https://figma.com/.../1-3

Claude builds static, pauses for your "looks good", then makes dynamic + tells you to sync. Same result, fewer messages.

📝 About brief files (optional, automatic)

Claude auto-creates a brief file at briefs/<section-name>.md on every build, so there's a paper trail of what was built and why. You don't have to write these yourself. If you DO want to write one before building (because the section has complex rules or you want to plan ahead), copy briefs/_template.md and fill in 5-15 lines — Claude will read it instead of auto-creating.

What to type — common prompts

Copy-paste these. Replace <section> with the section name (like home-hero), <url> with the Figma link.

@<section> <figma-desktop-url> <figma-mobile-url>

Two Figma links = Claude pixel-matches BOTH desktop and mobile exactly. One link = Claude builds desktop AND auto-makes it responsive for tablet/mobile (it tells you what assumptions it made so you can confirm). No brief file needed — Claude auto-creates one.

Everything after the Figma links becomes extra deviations. If a brief file already exists, both apply (merged additively, never replaced).

Build static + dynamic in one shot

Claude builds static, pauses for your "looks good", then converts to dynamic.

Just convert an already-built static to dynamic

Run this after you've confirmed the static looks good.

Compare live page vs Figma

Claude takes a screenshot of your live page and compares it to the Figma frame, reports what's off.

Fix something specific

Just describe what's off in plain English. Be specific (which section, what's wrong, what it should be).

If the design uses a color that's not in your project's palette and you'll use it more than once.

Before you hand off to a client / staging / live, run this. Claude checks accessibility, performance, typos, broken links, ACF sync state.

📝

Brief files (optional — auto-created)

You don't have to write briefs.

Claude auto-creates briefs/<section-name>.md on every build, using the Figma data + your chat context. The brief file exists for the paper trail (retros, handovers, future-you), not because you need to write it first.

If you WANT to write one upfront — for a section with complex behavior, multiple deviations, or because you want to plan ahead — go for it. 5-15 lines is plenty. Don't overthink. The Figma link does most of the work.

Minimum useful brief:

Section: Services Grid
Template owner: Default flexible (one of many on inner pages)
Figma desktop: https://figma.com/design/abc?node-id=5-1
Figma mobile: https://figma.com/design/abc?node-id=5-2

Behavior: 3 cards on desktop, stack vertical on mobile.

Deviations: per-card image not bg image (matches what designer agreed).

Things to include (if you write one):

  • Section name + which template (Homepage or default flexible)
  • Figma URLs (desktop + mobile if both exist — designer says yes? include both)
  • Content blocks (heading, body, image, CTA, list of cards, etc.) — Claude will probably figure these out from Figma but listing helps
  • Behavior — anything weird (sticky? modal? scrolling animation? mobile reorder?)
  • Deviations — anything you want different from Figma (rare; usually leave blank)

Things to NOT include:

  • Code (Claude writes the code)
  • HTML / CSS / Tailwind classes (Claude picks these)
  • ACF field types (Claude designs these in phase 2)
  • Long descriptions (the Figma link is your description)

When something looks weird — try these first

Most "bugs" are actually small things you can fix yourself in 30 seconds. Try this list before calling your dev.

🔁 The page looks unchanged after Claude said it built something

Hard refresh the browser. Ctrl+Shift+R (Windows) or ⌘+Shift+R (Mac). The browser caches old CSS — hard refresh forces it to fetch new.

🎨 Tailwind colors / spacing aren't applying (looks unstyled)

The Tailwind watcher might have stopped. In a terminal (ask your dev to do this if you're not comfortable), run npm run watch in the theme folder. Leave it running.

Tell-tale sign: the page is there but completely without colors / spacing / fonts. Like a 1996 website.

📋 ACF fields don't appear in the WordPress page editor

You forgot to sync. Go to WP Admin → Custom Fields → Field Groups → click "Sync changes" (blue notice at the top).

🖼 An image isn't showing

Two possibilities:

  1. The ACF image field is empty — go to the page editor, scroll to that section's fields, upload an image.
  2. Claude saved the image but with a different file name. Tell Claude: "The hero image isn't loading — what's the path?"

📐 Section is there but looks wrong (spacing / colors / fonts off)

Tell Claude exactly what's off. The more specific, the faster the fix:

The h2 in services-grid is way too big on mobile — Figma says 28px but I see 48px.
The card spacing is wrong — Figma has 24px between cards, the page has like 4px.

🎨 Claude stops mid-build and says "Figma is disconnected"

This is good behavior, not a bug. Claude is built to refuse to guess from screenshots because that produces sections that drift visually from each other. Quick fix:

  1. Open Cowork Settings → Connectors → Figma → click "Reconnect"
  2. If that doesn't work, restart Cowork and reopen the project folder
  3. Come back to the chat and type: continue

Claude will resume the build right where it stopped — no work lost. See INSTALL-MCPS.md if reconnect keeps failing.

🤖 Claude says "tools unavailable" before you've started anything

The Figma connector was never enabled. Open Cowork Settings → Connectors → enable Figma → restart your Cowork session. See INSTALL-MCPS.md for the full steps.

🪦 The whole local site is broken (white screen of death)

Stop, call your dev. This is usually a PHP error and the message is hidden. Your dev can turn on debug mode and tell you exactly what broke.

📞

When to call your developer

Claude is good at building sections. Your developer is good at the stuff Claude shouldn't touch. Here's the split.

✅ Do these with Claude

  • Building any section from Figma
  • Tweaking visuals (spacing, colors, fonts)
  • Converting static to dynamic (ACF)
  • Adding new sections or pages
  • Comparing live to Figma (/pixel-check)
  • Pre-deploy checks (/ship-check)
  • Editing text/images via WP Admin (no Claude needed)

⚠️ Call your developer for these

  • Setting up the WordPress install + theme + plugins
  • White screen / PHP fatal errors
  • Database errors
  • Deploying to staging or live (uploading to the host)
  • Plugin licensing / SSL certificates / DNS
  • Things that involve a terminal / command line beyond npm run watch
  • Forms not sending email (SMTP config)
  • Anything where Claude says "this is outside the .claude/ scope"

📨 How to ask your dev for help

Give them four things so they don't have to interrogate you:

  1. What you were trying to do ("I asked Claude to build the testimonials section")
  2. What you expected ("Section should appear like in Figma")
  3. What actually happened (paste the exact error message + Claude's response if any)
  4. The page URL or section name involved

Without these four, your dev does 5 messages of detective work before they can help. With them, they help in 1.

✅

Prompts that work great

Specific. Concrete. Tells Claude what you see and what you want.

✓ Specific 
The "Book Now" button in services-cta should be sticky-bottom on mobile only (showing in Figma frame 8-3 lower-right) — currently it scrolls with the page.
✓ Has Figma reference 
In Figma the card border-radius is 16px (https://figma.com/.../?node-id=4-12) but I see square corners on the live page.
✓ Compares to expectation 
After "/build services-grid" the section appears but the 3-card grid isn't stacking on mobile — they're all overlapping. Should stack vertically with 24px gap as Figma shows.
✓ Acknowledges what Claude did 
The deviations Claude flagged on testimonials (using arbitrary 73px padding) look fine — approved. Now let's make it dynamic.
❌

Prompts that waste time

Vague. Missing context. Claude can't read your mind — be specific.

✗ Too vague 
it looks wrong

→ Better: "The h2 in home-hero is showing as 24px but Figma is 48px. Fix it."

✗ Asks Claude to read your mind 
make the hero better

→ Better: "In the hero, make the CTA button larger (Figma shows it at 56px tall, currently it's ~44px)."

✗ Multiple sections in one message 
build hero, services, and testimonials all together

→ Better: build them one at a time. Trust the workflow. The whole template is designed around section-by-section.

✗ Skipping the static phase 
build and make dynamic in one go

→ Better: /build <section> <url> — Claude builds static, pauses for your approval, then makes dynamic. The pause is essential.
